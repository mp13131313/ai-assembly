### FU#49A v2 — Pass 4b `quality_criteria` field-tied 3-dim scaffold (commit 0ca02f5)

**Problem:** v1 (LANDED 2026-04-26 in `e6fc634`, reverted in `9480d3a`) used procedural mandates: *"5 fidelity criteria + 6th generativity criterion + 7th strain-marking criterion."* Empirical observation: procedural mandates produced checkbox satisfaction — the model added a "generativity" criterion that was vague and a "strain-marking" criterion that was specific but neither was load-bearing for "Could this be mine?" testing.

**Solution after 15 standalone exemplar tests:** swap the BLOCK 2 `quality_criteria` field-spec to a 3-dim field-tied scaffold:

```
- quality_criteria: 3-5 specific, testable criteria. Each criterion
  tests 1-n card fields by name (single field, or compounded with
  and/or logic). The criteria collectively should answer "Could
  this be mine?" across reasoning, voice, and form. Available
  load-bearing fields (consult as needed):

    - REASONING: reasoning_method, constitution, concept_lexicon
    - VOICE: rhetorical_mode, characteristic_moves,
      register_and_tone, metaphorical_repertoire
    - FORM: medium, characteristic_output_structure,
      aesthetic_qualities, stance_tendency
```

**Empirical result on Plato re-run:** 5 criteria, all field-named, all testable, criterion #5 spontaneously emerged as trace-preservation: *"Could a careful reader walk away with the question sharpened... has the piece done the elenctic work, or has it merely reported that such work could be done?"* — without any procedural mandate triggering it. The "consult as needed" framing gave the LLM the right freedom: use the field list as scaffolding, extend to non-listed fields when warranted.

**What was NOT re-landed in v2** (deliberate): trace-preservation in `relationship_to_detailed_response` and length-variance in `length_and_format_constraints` (both part of v1 and reverted alongside). The hypothesis from the 04-28 revert was that cumulative additions caused texture loss; v2 deliberately landed only the ONE most load-bearing piece. Trace-preservation came back organically through the field-tied criteria framing anyway.

### FU#54 — Smoke test runtime-fidelity refactor FILED (commit 5da5434)

Filed as POST-ATHENS. Three drift layers identified between Pass 7b's smoke test and the actual Voice Pipeline runtime architecture:

1. **Card delivery:** 7b puts the full card as a JSON blob in the user prompt with the wrapper *"COMPLETE PERSONA CARD: {json}"* + a system prompt that is the 7b instruction. Voice Pipeline runtime puts the card AS the system prompt rendered as structured headers (`IDENTITY` / `CONSTITUTION` / `REASONING METHOD` / `ENGAGEMENT`) + an XML-tagged closing instruction. The operator's manual chat-test workflow (paste `03_chat_system_prompt.json` as Claude project custom instructions, ask a question) matches runtime; 7b does not.

2. **Field set:** 7b uses ALL 35 generated fields in one call. Real runtime: Step 1 uses foundational + reasoning fields (1-21 minus smoke_test_chains); Step 2 uses voice + artifact fields (22-34); Step 3 uses both. 7b conflates Step 1 + Step 2 into a single shot.

3. **Step boundaries collapsed:** 7b produces "provocation→response chains" — both reasoning AND artifact in one call. Real runtime: Step 1 produces `detailed_response` (private); Step 2 reads Step 1's output + voice/artifact fields and produces the public artifact via family-of-forms selection.

**Operator's preferred design** (filed 2026-04-29): restructure 7b into 6 LLM calls — 1 provocateur call (no card) generating 3-5 provocations; 3-5 standalone voice-response calls (card as system prompt per Voice Pipeline runtime spec, single provocation as user prompt). Mirrors per-call shape of actual runtime.

**Why post-Athens:** chat-test is sufficient runtime-fidelity gate for Athens panel (empirically validated on Plato 04-26). Pre-Athens prompt restructuring + 7c calibration shift carries tail risk.

## Architectural decisions ratified this session

### "Card as system prompt at runtime" is correct

Operator question raised mid-session: should the runtime put the persona card as system prompt (current Voice Pipeline spec + chat-test workflow) or as JSON blob in user prompt (what Pass 7b does)? Three reasons card-as-system-prompt is right at runtime:

1. **Anthropic prompt caching:** at 10 voices × 30 briefings/night, system prompt caches per voice; user prompt varies. ~5-10× cost difference at panel scale.
2. **Training emphasis:** Claude's persona-following weights system-prompt content more heavily than user-prompt JSON. Card register-discipline lands where it's meant to.
3. **Empirical:** the Plato 04-26 chat test (paste chat_system_prompt as Claude project custom instructions) is what gave the validation that shipped Plato to Phase L sign-off. Don't move what's working.

Pass 7b doesn't match — and that's filed as FU#54.

### Variance design: single-form-family per voice + within-form variance is the architecture

Card v2.1 §H specified family-of-forms emission (default form + 3-6 variations per voice). Landed `e6fc634` (2026-04-28). Reverted `9480d3a` (2026-04-28 evening) when chat-test showed cumulative additions degraded texture. 04-29 standalone test on Plato confirmed: even lighter "permit but don't mandate" version produced texture loss for Plato (lost named scenes). Plato's variance lives in 6 within-form axes (scene, interlocutor, ending mode, move-emphasis, length-within-band, stance amplitude).

Cleopatra is the real test of cross-form variance (FU#55 trigger).

### Family-of-forms in spec ≠ family-of-forms in prompts

Card v2.1 §H docs section says family-of-forms is the design. Pass 4b prompts after 04-28 revert say "one phrase" for `medium`. The spec doc is **forward-looking aspirational** for voices with multi-form corpora; the implementation is **single-form-locked** until Cleopatra empirically validates the §H attempt.
