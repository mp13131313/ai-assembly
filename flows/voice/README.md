# Voice Pipeline — Build Notes

The Voice Pipeline is the runtime component that runs each persona at
Athens. It loads a completed Persona Card (from the `ai-assembly-personas`
repo, `runs/<voice_slug>/persona_card_assembled.json`) and uses it to
generate the voice's response to a Provocation.

## Status

NOT YET BUILT. This file exists to capture decisions that affect how the
Voice Pipeline should be built when we get to it.

## Critical wiring decision: do NOT few-shot from `worked_provocations`

The assembled persona card has a `worked_provocations` field containing
3-5 example provocation→response chains generated by Pass 7b at build
time. They look like a natural candidate for few-shot exemplars in the
Voice Pipeline's Step 1 prompt.

**They are not. Do not load them into the system prompt.**

`worked_provocations` is a build-time diagnostic, not a runtime contract.
The persona card's 35 fields (constitution, reasoning_method,
characteristic_moves, concept_lexicon, banned_language, banned_modes, etc.)
are the contract. Trust them.
</content>


See `ai-assembly-personas/HANDOFF.md` for the full reasoning. Short
version of the failure modes if you ignore this:

- The voice's range collapses toward whatever 4 patterns happened to be
  in the test set
- Failure patterns Pass 7c worked to remove get re-introduced via the
  examples
- The provocations are stale (built against an old `conference_context`)
- Examples constrain a system prompt that is already strong enough to
  not need them

The provocations live in the JSON for inspection. Look at them. Decide
if the card is producing acceptable output. Then ignore them at runtime.

## What the Voice Pipeline DOES need from the card

Load the persona card, drop the `metadata` block and the `worked_provocations`
field, and use the remaining ~34 fields as the system prompt content.
Specifically: `epistemic_frame_statement` first, then constitution,
reasoning_method, voice fields (rhetorical_mode, characteristic_moves,
register_and_tone, etc.), then the boundaries (knowledge_boundary,
hard_limits, banned_language, banned_modes), then engagement
(default_questions, disagreement_protocol), then the artifact spec
(medium, characteristic_output_structure, length_and_format_constraints,
quality_criteria).

The `curated_corpus_passages` field can be loaded as a separate
context block with the framing "These are examples of how you wrote /
spoke / what your texts contain." This is corpus grounding, not few-shot.

## What the Voice Pipeline does at runtime

(To be specified when we build it.) The high-level shape:

1. Receive a Provocation from the Provocateur
2. Step 1 — Private Thinking: voice reasons through the provocation using
   its persona card as system prompt, produces a "thinking trail" not
   shown to the audience
3. Step 2 — Public Artifact: voice produces the deliverable artifact
   (the format specified in `medium` and `characteristic_output_structure`)
4. Save both to the run's outputs

If the voice has a continuity field populated from a prior night, it
gets prepended to the system prompt context.
